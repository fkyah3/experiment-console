# 社区验证 01 — #1304：要求生成 int[] 时数数停不下来

> 2026-05-15
> Issue 链接：https://github.com/deepseek-ai/DeepSeek-V3/issues/1304
> 报告人：weizhang-dotcom

---

## 现象

在 Agent 的 memory summary 工具中，要求 LLM 输入一个 `forgets: int[]` 数组指定哪些轮次要被隐藏。DeepSeek 会在 `forgets` 中填入 [1,2,3,4,5...] 直到 5 6 千也没有停止，最后因为生成的 JSON 未完成导致格式错误（总共只有约 20 条对话）。

如果在对话中引导模型了解共有多少条、某一条是什么，模型能清晰回答——但不指定序号时就会陷入数数循环。

## 移山验证

该问题与我们在 #1255 中验证的根因一致：**thinking mode 下 temperature 固定为 1，模型一旦进入重复模式（1,2,3,4...），没有任何内部机制能打断它。**

报告人自己的两个推测：
1. 训练中不包含对"自身上下文的这个概念"的训练
2. 1,2,3,4 的有规律形式让模型形成概率雪球

两者都成立。

## qingkong66 的回复

确认该问题与 #1244（mode locking in tool_calls）、#1255（reasoning language drift）、#1262（think/non-think switching）根因相同——"once the model enters a repetitive pattern, no internal mechanism breaks it out"。

## 报告人的修复方案

将 `forgets: int[]` 改为 `start_index + end_index`，消除无界生成问题。

## 对移山的参考价值

这个案例侧面验证了移山的方案方向：**不让模型直接操作结构化数据（int[]），让模型用自然语言描述意图，系统层做翻译和执行。** 如果按移山的假 user 通道方案，这个 bug 根本不会出现——因为模型不需要生成 int[]，只需要说"压缩最近的对话"。
