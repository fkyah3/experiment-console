# 社区验证 03 — #1311：结构化约束让生成稳定性碾压自由发挥

> 2026-05-15
> Issue 链接：https://github.com/deepseek-ai/DeepSeek-V3/issues/1311
> 报告人：linyy750911（老林）

## 现象

非程序员老林发明了一套"Code Bagu（代码八股）"结构化注释协议，强制 AI 按固定格式输出代码。对比实验数据：

| 指标 | 自由发挥（Claude） | 八股约束（DeepSeek） |
|:-----|:------------------|:---------------------|
| 校验违规 | 34 个 | 0 个 |
| 健康检查器思维链 | 8000+ 字死机 | 3.2 秒完成 |
| 写入拦截率 | N/A | 50% → 0%（收敛） |

## 移山验证

老林的问题和我们面对的是同一类问题（"模型太聪明，聪明到会跑偏"），但解决路径不同：

| | Code Bagu | 移山 |
|:-|:---------|:------|
| 约束方式 | 结构化注释模板（prompt 层） | 假 user 通道（架构层） |
| 适用范围 | 固定模板代码生成 | 通用 tool calling 场景 |
| 优势 | 简单、可移植、零架构成本 | 彻底消除协议层副作用 |
| 局限 | 只适用于固定模板场景 | 需要搭整套基础设施 |

## 老林的核心发现

> "不是模型竞赛，是工程方法竞赛。"

这个判断和移山的立场一致——DeepSeek 在结构化任务上的顺应性优于 Claude，配合好的约束方法可以做出稳定可靠的东西。

## 对移山的参考价值

老林的对偶规则引擎（🔒/🔓 strict/coverage 两种模式）对移山的后续开发有参考价值——某些约束不可妥协（资源 1:1），某些约束可松弛（异常处理）。这个约束等级化的思路移山还没系统化。
