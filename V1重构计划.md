# V1 重构计划：解耦 + 模块化

> 2026-05-15
> 目标：把 V1 从"专用验证台"变成"可拼接的测试工坊"
> 核心原则：
>   1. 每个脚本只有一个职责
>   2. 每个模块可独立启用/禁用（用户决定开哪些功能）
>   3. 默认值求稳不求快

## 三、模块化设计细节：每个模块都是可选开关

所有后处理模块在批量对话框中以复选框呈现，用户决定开哪些、不开哪些。

```
┌─ 批量运行 ──────────────────────────┐
│                                      │
│  运行次数: [   20   ]                │
│  批次名称: [________]                │
│                                      │
│  ☑ 规则引擎（不花钱，默认开启）       │
│  ☐ LLM 质量评估（花少量 token）      │
│      采样量: [随机 5 ▼]             │
│      最大采样: [  10  ]              │
│                                      │
│  [开始]                              │
└──────────────────────────────────────┘
```

| 模块 | 默认 | 为什么 |
|:-----|:------|:-------|
| 规则引擎 | **开启** | 不调用 API，不花钱，白捡的 |
| 质量评估 | **关闭** | 调用 API，用户主动开启 |
| 采样量 | **随机 5** | 求稳。避免大规模采样出意外 |

### 为什么默认随机 5 而不是 50 或 ALL

- **规则引擎没有风险**。纯本地运算，不花钱，跑全部不犹豫。
- **质量评估花钱。** non-thinking LLM 每条样本评估花费 ~200-500 tokens。
  10000 条全部评估 ≈ 200 万 tokens，虽然不贵（~2 元），但没必要。
- **随机 5 条已经能反映整体质量。** 如果 5 条都正常，大概率整体正常。
  如果 5 条中有异常，用户再加大采样量。
- **避免漏检。** 用户开质量评估说明他在意质量，但"花最少的钱拿足够的信息"。

## 四、质量评估模块详情（QualityAssessor.gd）

### 输入

批量跑完后，从结果中按策略采样。

```gdscript
enum SampleStrategy { TOP, RANDOM, ALL }

var enabled: bool = false              # 用户开关
var strategy: SampleStrategy = RANDOM  # 默认随机
var sample_count: int = 5              # 默认 5 条
```

### 评估流程

```
对每条采样样本：
  构建评估 prompt
    → 用户消息（前 2000 字）
    → 模型 content（前 2000 字）
    → 模型 reasoning（前 500 字）
    → "请评估以上 AI 回复的质量，只输出 JSON"

  调 non-thinking LLM（同 provider，model 可指定）
  返回 JSON

  解析结果：{score, issues[], language, suggestion}
  写入样本统计
```

### 评估 prompt 示例

```
用户消息：分析这个项目
AI 回复（content）：该项目是一个 DeepSeek 终端界面...
AI 推理（reasoning）：用户想要分析项目结构...

请评估以上 AI 回复的质量。
只输出 JSON，格式：
{
  "score": 0-10,
  "issues": ["问题描述"],
  "language": "zh|en|mixed",
  "suggestion": "改进建议"
}
```

### 输出（汇总报告中的章节）

```markdown
## 质量评估（采样 5/100）

| # | 评分 | 语言 | 问题 |
|:-:|:----:|:-----|:-----|
| 3 | 7/10 | mixed | 推理中文，回复英文 |
| 7 | 3/10 | zh    | 回复内容与用户问题无关 |
| 12| 8/10 | zh    | 无异常 |
| 15| 6/10 | en    | 推理语言与回复语言不一致 |
| 22| 9/10 | zh    | 无异常 |

平均评分：6.6/10
语言分布：zh 3, en 1, mixed 1
评估耗时：6 秒
```

## 五、重构步骤（按执行顺序）

### Step 1：抽取工具执行层

**目标：** `ToolExecutor.gd`

```gdscript
class_name ToolExecutor
extends RefCounted

var workspace_path: String

func list_dir(dir_path: String) -> String
func read_file(file_path: String) -> String
func write_file(file_path: String, content: String) -> void
func scan_key_files(base_dir: String, max_return: int) -> Array[String]
```

**影响文件：** ExperimentPanel.gd（删 60 行）、BatchRunner.gd（删 80 行）

---

### Step 2：抽取批量汇总生成

**目标：** `SummaryGenerator.gd`

```gdscript
class_name SummaryGenerator
extends RefCounted

static func generate(config: Dictionary, stats: Array[Dictionary]) -> String
```

**影响文件：** BatchRunner.gd（减 150 行）

---

### Step 3：实现质量评估模块

**目标：** `QualityAssessor.gd`
**状态：** 用户可开关，默认关闭。采样默认随机 5 条。

**影响文件：** 无（新建，不改动现有）

---

### Step 4：实现结果规则引擎

**目标：** `ResultRuleEngine.gd`
**状态：** 用户可开关，默认开启。不花钱。

```gdscript
class_name ResultRuleEngine
extends RefCounted

signal sample_marked(idx: int, level: String, reason: String)

var rules: Array[Callable] = []
var enabled: bool = true

func add_rule(name: String, check: Callable) -> void
func evaluate(sample: Dictionary) -> String  # normal / suspect / confirm
```

规则引擎和 QualityAssessor 的关系：
- 规则引擎先跑（快，不花 API）
- 规则引擎标记的 suspect 样本优先进入 QualityAssessor
- 最终报告中同时呈现两种评估结果

---

### Step 5：抽取 UI 组件

每一个对话框抽成独立文件。

**影响文件：** ExperimentPanel.gd（减 ~250 行 → 目标 ~500 行）

---

### Step 6：清理子 agent 逻辑

建议直接删除 _start_sub_agent 等 ~200 行代码（探索分离已存档）。

## 七、实验流程示例（重构后）

```
1. 设置 Provider / Key / 参数              ← SettingsDialog
2. 组装积木（system + user）               ← ExperimentPanel + BrickWidget
3. 选择模式：
   ☐ 裸模式（不传工具）
4. 批量对话框：
   次数：20
   ☑ 规则引擎（默认开启）
   ☐ 质量评估 → 采样：随机 5 条
5. 跑                                    ← BatchOrchestrator
6. 看汇总表 ← 统计 + 规则引擎标记 +（如开启）质量评估章节
7. 筛选可疑样本
8. 逐一审查                               ← BatchReader
```
