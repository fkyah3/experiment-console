# 工程复盘：工具调用调试全记录

> 2026-05-09
> 涉及：ExperimentPanel.gd、DeepSeekStreamClient.gd、APIBuilder.gd

---

## 一、问题

`deepseek-v4-flash` 在收到包含 `tools` + `tool_choice: "auto"` 的请求后，只返回 `reasoning_content`，不返回 `tool_calls`。

## 二、调试过程

### 第 1 轮：试错

直觉猜测：模型不认识工具定义、tool_choice 格式不对、thinking 模式干涉。

修复：
- `build_body_dict()` 补了 `tool_choice: "auto"`
- thinking 模式换成"官方预设"
- 消息结构微调

结果：❌ 仍不触发。

### 第 2 轮：分析层介入

分析层对比 DeepSeek-TUI 代码后发现两个差异：

1. **reasoning_content 发送方式**：我们旧代码把 reasoning 合并到 content 发送；DeepSeek-TUI 始终用 `reasoning_content` 独立字段
2. **缺少 sanitizer**：DeepSeek-TUI 发送前遍历所有 assistant 消息，缺 `reasoning_content` 的补 `(reasoning omitted)`

修复：
- 改 `build_api_messages()`：有 reasoning 就发 `reasoning_content`，不合并
- 加 sanitizer：`_do_send()` 中补 `(reasoning omitted)`

结果：❌ 仍不触发。方向偏了。

### 第 3 轮：发现真正根因

分析层用 Python 精确复刻请求体，发现：
- **DeepSeek SSE streaming API 全程不发 `finish_reason` 字段**
- 旧代码依赖 `finish_reason == "tool_calls"` 来设置 `_had_tool_calls`
- 这行代码永远不命中 → `_had_tool_calls` 永远 `false`
- 即使 `_tool_call_buf` 里有完整数据，发的是 `stream_finished` 而不是 `tool_calls_done`
- `ExperimentPanel` 只处理了 content/reasoning，tool_calls 被静默丢弃

### 关键转折点

分析层给出证据链：
- 精确复刻实验请求体 → Python 请求 → API 返回了 `list_dir(".")` 工具调用
- 原始 SSE 事件流（46 个事件）→ `finish_reason` 全程为 `None`
- 3 次重复 → 100% 触发

说明：**API 本身是正常工作的，问题在我们解析 SSE 的逻辑上**。

## 三、根因

| 层 | 问题 |
|:---|:------|
| 表象 | 模型不调工具，只返回 reasoning |
| 直接原因 | SSE 解析中 `finish_reason` 检测不命中 |
| 根本原因 | DeepSeek streaming API 全程不发 `finish_reason`，但代码假设它会在最后一个 chunk 里发 `"finish_reason": "tool_calls"` |
| 深层次 | 先入为主地用了 OpenAI 的 SSE 规范来写 DeepSeek 的解析器 |

## 四、修复（3 处）

见 [DeepSeekStreamClient.gd](file:///e:/agent/MountainShift/experiment-console/scripts/DeepSeekStreamClient.gd)：

| # | 位置 | 改前 | 改后 |
|:-:|:-----|:-----|:-----|
| 1 | `_parse_event()` | 用 `finish_reason == "tool_calls"` 设 `_had_tool_calls` | **删除** |
| 2 | 同上，`delta.has("tool_calls")` 块内 | 无 | **加** `_had_tool_calls = true` |
| 3 | STATUS_DISCONNECTED + [DONE] 两处 emit | `if _had_tool_calls` | `if not _tool_call_buf.is_empty()` |

配套修复：
- `APIBuilder.gd`：reasoning_content 独立发送 + tool_choice: auto
- `ExperimentPanel.gd`：sanitizer 兜底 + `_tc_max_rounds` 改为 `@export var`

## 五、教训

### 5.1 信号选错，全盘皆输

依赖 `finish_reason` 判断工具调用，但这个信号在 DeepSeek 的 SSE 中不存在。

换一个思路：**不依赖 API 的信号，直接从自己已经收到的数据判断**。`_tool_call_buf` 是否有数据，是比 `finish_reason` 更可靠的指标。

### 5.2 三个方向失败才换路是对的

第 1 轮改 body 参数、第 2 轮改消息结构——这两个方向都错了。到第 3 轮分析层给出 Python 精确复现 + 原始 SSE 事件流时，才发现是解析层的问题。

按铁律：同一个根因方向试 3 次失败就应该换路——我们这次在方向选择上绕了久了一点，但最终收敛对了。

### 5.3 工具调用的核心检查清单

对于 SSE 流式 API 的工具调用支持，必须验证：

- [x] API 本身是否支持 tools（Python 裸请求验证）
- [x] 请求体中 tools 是否格式正确
- [x] 流式事件中 tool_calls 的 delta 格式
- [x] 解析层是否能正确检测到 tool_calls 到达
- [x] 解析层在流结束后是否发了正确的信号
- [x] 接收方是否处理了该信号
