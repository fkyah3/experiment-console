# 中控层通报：工具调用修复与社区沟通要点

> 2026-05-09
> 用途：中控层与 DeepSeek 社区沟通时的参考

---

## 一、背景

Experiment Console 项目验证 DeepSeek V4 Flash/Pro API 的 tool calling（function calling）。复现过程发现：API 本身支持工具调用，但我们的客户端因 SSE 解析逻辑错误，导致工具调用被静默丢弃。

## 二、核心发现

### DeepSeek SSE streaming API 不发 finish_reason

这是本次调试的最大发现。与 OpenAI 的 SSE 规范不同，DeepSeek 的 `stream=true` 模式在最后一个 delta chunk 中**不包含 `finish_reason` 字段**。

对比：

| API | 最后一个 delta chunk |
|:----|:--------------------|
| OpenAI | `{"choices":[{"delta":{},"finish_reason":"stop"}]}` |
| DeepSeek | `{"choices":[{"delta":{}}]}` — **无 finish_reason** |
| OpenAI 工具调用 | `{"choices":[{"delta":{"tool_calls":[...]},"finish_reason":"tool_calls"}]}` |
| DeepSeek 工具调用 | `{"choices":[{"delta":{"tool_calls":[...]}}]}` — **无 finish_reason** |

结论：**DeepSeek 的 SSE 事件不包含 finish_reason，依赖 finish_reason 的客户端代码会永久失效**。

### 正确的检测方式

不应依赖 API 信号，应依赖本地状态：

```python
# 错误：依赖 API 的 finish_reason
if finish_reason == "tool_calls":
    process_tool_calls()

# 正确：依赖本地累积的数据
if len(tool_call_buffer) > 0:
    process_tool_calls()
```

### 工具调用已成功复现

修复后，`deepseek-v4-flash` + `thinking.enabled` + `reasoning_effort: high` 配置下：

- `list_dir` 工具：正常工作，模型能扫描目录结构
- `read` 工具：正常工作，模型能读取文件内容
- 多轮工具调用：正常工作，最多可配置 50 轮连续调用

## 三、给 DeepSeek 社区的建议

### 3.1 文档补充建议

DeepSeek API 文档中关于 stream=true 模式下 SSE 事件格式的部分（特别是工具调用场景）建议补充：

> **注意：DeepSeek 的流式响应不包含 finish_reason 字段。** 工具调用检测应基于 delta 中的 tool_calls 字段存在性，而非 finish_reason 的值。每个包含 tool_calls 的 delta chunk 是自包含的（含 id、function.name、function.arguments），客户端应在收到 tool_calls 数据时即时累积，在收到 [DONE] 或连接关闭时触发处理。

### 3.2 其他客户端注意

使用 OpenAI SDK 封装 DeepSeek API 的客户需要注意：

1. `finish_reason` 在 stream=true 时始终为 null（连 `"stop"` 也没有），不会在最后一个 chunk 出现
2. `usage` 信息需要额外处理——它在独立的 JSON 事件中返回（`"choices":[],"usage":{...}`），需单独捕获
3. `thinking` 模式下的 `reasoning_content` 格式与 `content` 一致（都是 delta 字段），但**不会与 tool_calls 共存于同一 delta chunk**

### 3.3 工具调用的 reasoning_content 处理

当 API 返回工具调用时：
- 该轮次的 `reasoning_content` 出现在 tool_calls delta 之前的独立 chunk 中
- 多轮对话时，上一轮有工具调用的 assistant 消息**必须回传 reasoning_content**（否则 API 返回 400）
- 建议实现 sanitizer：发送前遍历所有 assistant 消息，缺少 reasoning_content 的补 `"(reasoning omitted)"`
