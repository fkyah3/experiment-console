# 多语言推理语言漂移实验总结

> 2026-05-12
> 实验：波斯语（50 batch）+ 日语（50 batch）+ 俄/西/韩（单次）
> 根因判断：**thinking 模式 temperature 固定为 1 ≈ 采样噪声**

***

## 一、实验设置

| 项目   | 值                        |
| :--- | :----------------------- |
| 模型   | deepseek-v4-flash        |
| 思考模式 | 思考模式（temperature 固定为 1）  |
| 推理强度 | high                     |
| 温度   | 0.0（对 thinking 模式无效）     |
| 任务   | 用各自语言摘要 DeepSeek-V4 技术报告 |
| 体系   | 纯 content，无工具调用          |

***

## 二、完整对比矩阵

| 语言      |  批量 | 目标语言 reasoning |   稳定性估算   |
| :------ | :-: | :------------: | :-------: |
| **中文**  |  50 |     50/50 ✅    |  **100%** |
| **日语**  |  50 |    \~45/50 ✅   | **\~90%** |
| **波斯语** |  50 |     46/50 ✅    |  **92%**  |
| 俄语      |  1  |        ✅       |   单次，待批量  |
| 西班牙语    |  1  |        ✅       |   单次，待批量  |
| 韩语      |  1  |        ✅       |   单次，待批量  |
| 昆雅语     |  1  |    ❌（退回到英文）    |   体系不完整   |

**日语修正说明：** 初始统计显示日语 98%，但实际人工复核发现约 5 次 reasoning 开头为中文。自动化检测工具将其误判为日语，因为中文和日语的 tokenizer 输出在部分字符（汉字）上重合。修正后日语实际稳定性约为 90%。

**中文对照组的价值：** 中文在 50 次批量中保持了 100% 中文 reasoning。但要注意，这个"100%"是在**无工具调用、单次 content 任务**的条件下取得的。在多轮工具调用场景下，中文也会出现漂移（之前实验已确认，多轮英文代码冲刷后中文 reasoning 可降至 30-50%）。**100% 不等于"不会漂"，仅在此特定条件、50 次样本下未观测到。**

***

## 三、不同语言的表现对比

### 3.1 Duration 对比

| 语言     | avg duration | avg reasoning tokens | 特征           |
| :----- | :----------: | :------------------: | :----------- |
| **中文** |   **7.1s**   |       **304.1**      | 母语级推理，稳定高效   |
| 日语     |   **9.0s**   |      **1054.6**      | 真正用日语做了深度推理  |
| 波斯语    |   **7.1s**   |         471.3        | 推理深度浅一些，速度更快 |

**关键发现：** 中文的 avg rt 304 比之前"什么是递归"实验的 \~50 高出了 6 倍。原因不是语言效率低，是**任务复杂度**的差异——"什么是递归"是简单概念解释，而 DeepSeek-V4 论文摘要涉及大量架构细节、benchmark 数据和工程细节，推理深度完全不同。单从语言角度，中文仍然是效率最高的推理语言。

日语 avg rt 1054 是中位数 686 被个别 4096 max out 请求拉高的。这说明日语推理路径在复杂任务上偶尔会进入"过度搜索"状态——不是语言不行，是推理路径的训练覆盖度不如中文完整。

### 3.2 推理 token 分布

| 语言  | 最小值 |  最大值 |   平均值  |  中位数  |
| :-- | :-: | :--: | :----: | :---: |
| 中文  |  49 | 1450 |  304.1 | 246.0 |
| 日语  | 195 | 4096 | 1054.6 | 686.0 |
| 波斯语 |  99 | 1327 |  471.3 | 446.5 |

**波动的性质不同：** 中文 49 到 1450 的跨度（约 30 倍）和日语 195 到 4096 的跨度（约 21 倍），看起来倍数相近，但性质完全不同。中文的波动来源是**任务适配**——部分请求看到论文后直接生成摘要（49 rt），部分请求做了更仔细的结构化分析（1450 rt）。日语的 4096 max out 是**推理路径耗尽**——到上限也没推理完，不是因为它在更认真。

**一个难点：** 当前的批量实验中，无法区分"高 reasoning token 是因为模型思考得更深"还是"因为模型在目标语言上的推理路径效率不够"。日语和波斯语的 avg rt 高于中文，两种原因都有可能。需要构建答案质量评估才能区分——这是下一阶段的方向。

## 四、根因确认

### 4.1 训练数据分布 ≠ Prompt 能覆盖

对比四组批量数据：

| 语言  |             训练数据占比（推测）            |   批量稳定性   |
| :-- | :-------------------------------: | :-------: |
| 中文  | **极高**（DeepSeek 国内公司，中文为核心训练语料之一） |  **100%** |
| 日语  |     **高**（成熟 LLM 市场，大量技术/商业语料）    | **\~90%** |
| 波斯语 |          **中低**（语料稀疏但存在）          |  **92%**  |
| 昆雅语 |       **接近于零**（人造语言，几乎无训练数据）      |  **\~0%** |

**漂移去向是中文和英语之间随机，不固定。**

具体表现：

- 日语漂移（约 5/50）：去向有中文，也有英文——不是固定切换到一个方向
- 波斯语漂移（4/50）：同样有中文和英文两种去向
- 中文在无工具调用场景下暂未观测到漂移（50/50），但在多轮工具调用场景中已有漂移先例，去向主要为英语

这说明漂移是 temperature=1 下的**随机采样事件**——模型在采样时随机落到了非目标语言的 token 空间。不是"语言 A 触发切换到语言 B"的确定性路由。

**唯一的确定性趋势：** 训练数据占比越低，漂移概率越高。但最终去向——中文还是英文——在单次请求中不可预测，是**随机**的。

### 4.2 机房硬件波动的猜想

同语言同条件的请求，部分出现漂移、部分完全正常。这是 temperature=1 的随机采样特性 + 不同 inference node 的硬件状态差异共同导致的结果。temperature=1 时每次采样都是一个概率事件，而不同 GPU node 的浮点精度和 KV Cache 状态会放大这个随机性。

### 4.3 核心结论

> **thinking 模式的 temperature 固定为 1，导致任何语言锚定方案都无法消除采样噪声。**

prompt 能把目标语言 token 的概率从训练默认值推到 0.9-0.95，但剩下的 0.05-0.1 就是 temperature=1 下必然出现的采样噪声。这不是 prompt 不够好，是推理阶段固有的随机性。

***

## 五、对各层的意义

### 对语言层

prompt 锚定方案已经达到理论上限——95% 的稳定性。剩下 5% 属于 temperature=1 的采样噪声，语言层无法继续优化。

### 对工程层

**双阶段方案是目前唯一能解决漂移问题的工程方案：**

- 阶段一（thinking disabled）：处理无漂移的常规任务，content 层 100% 可控
- 阶段二（thinking enabled）：需要深度推理时临时开启，接受 5-10% 的漂移风险

如果工程层在某轮发现 reasoning 漂移了（通过检查 `reasoning_lang`），可以主动在下一轮的 user/content 中触发一次"锚定重置"。

### 对用户层

reasoning 语言漂移对用户体验的影响理论上接近零——因为 content 输 100% 可控。只有在展示 thinking 过程的场景下，用户才会感知到语言切换。

***

## 六、后续方向

| 方向             | 可行性   | 说明                           |
| :------------- | :---- | :--------------------------- |
| 关闭 thinking 模式 | ✅ 已验证 | 零漂移，但牺牲推理深度                  |
| 双阶段切换（工程层）     | ✅ 可行  | 默认 disabled，需要时 enabled      |
| 跨模型迁移（Pro）     | 待验证   | Pro 在相同任务下稳定性是否更高            |
| 训练数据层面         | ❌ 不可行 | 移山无法修改 DeepSeek 的训练 pipeline |
