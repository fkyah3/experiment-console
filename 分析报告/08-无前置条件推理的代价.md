# 工程复盘：无前置条件推理的代价

> 2026-05-09
> 事件：工具调用不触发 — 调试过程回顾
> 类型：AI 协作模式复盘（元复盘）

---

## 一、事件回顾

### 表象

`deepseek-v4-flash` 收到正确格式的 `tools` + `tool_choice: "auto"` 请求后，只返回 `reasoning_content`，不返回 `tool_calls`。

### 调试过程

```
Round 1: 改 body 参数（加 tool_choice: auto、改 thinking 模式）
  → ❌ 不触发

Round 2: 分析层介入 → 改 reasoning_content 发送方式 + 加 sanitizer
  → ❌ 仍不触发

Round 3: 分析层 Python 精确复现 + 原始 SSE 数据
  → ✅ 发现根因：DeepSeek SSE 不发 finish_reason
  → 修 3 行代码，成功
```

### 时间线

从 Round 1 开始到发现真正根因，中间绕了两次弯路。弯路的核心原因是：**在缺失推理前置条件的情况下，硬往下推**。

## 二、问题诊断：缺失的推理前置条件

当时列出的缺失前置条件有 5 个：

| # | 缺失条件 | 当时的状态 |
|:-:|:---------|:-----------|
| 1 | 不知道原始 SSE 数据长什么样 | 实验报告的 response 是代码重建的，不是原始 API 返回 |
| 2 | 未验证 `"\\n\\n"` 分隔符在 GDScript 中的实际行为 | Python 测试显示字面量 `\n\n` 不匹配实际换行符 |
| 3 | 没有工作请求的对照样本 | 不知道 Trae/DeepSeek-TUI 发的实际请求体是什么 |
| 4 | 未验证 `finish_reason` 检测在真实数据下是否工作 | 不知道真实 SSE 事件中 finish_reason 的行为 |
| 5 | 未确认磁盘代码就是实际运行代码 | 不确定 Godot 是否重新加载了修改后的脚本 |

### 关键错误

**其中第 4 个条件就是真正的根因，但 AI 没有指出"这个条件最可疑"。**

AI 列了 5 个前置条件，说"这些我都不知道，所以我没法继续推理"。这是**正确的做法**——停下来，不硬猜。但在列条件时，AI 没有区分优先级，把同样重要的根因（finish_reason）和不重要的旁支（分隔符行为、代码刷新问题）混在一起列。

用户介入说"5 个前置条件没有一个需要继续追查"，然后直接指出根因是 `finish_reason`。

### 为什么 AI 没发现是最优先的

AI 的思维过程是：
1. "我看到了很多可能性"
2. "这些我都缺信息"
3. "所以我没法判断"

但用户（分析层）的思维过程是：
1. "我看到了很多可能性"
2. "但其中一个可能性有 Python 实验证据支持"
3. "所以这个最可疑"

**AI 的弱点：** 在信息不足时，AI 倾向于"退守到安全区"（列条件、承认不足），而不是"基于已有证据做带风险的判断"。这在这种精确调试场景中是有害的——**宁可猜错方向然后验证，也不要停在原地列条件**。

## 三、根因精度对照

### AI 认为的根因（Round 2）

> "build_api_messages() 中 assistant 消息的 reasoning 不应该合并到 content"

**实际：** 这是次要因素。即使修了，工具调用仍不触发。

### 真实根因（分析层）

> **DeepSeek SSE streaming API 全程不发 finish_reason 字段。**
> 旧代码依赖 `finish_reason == "tool_calls"` 来检测工具调用，永远不命中。

**实际：** DeepSeek 的 API 行为与 OpenAI 不同，但代码是按 OpenAI 的 SSE 规范写的。修了 3 行就解决了。

### 精度差异分析

| 维度 | AI 推测 | 真实根因 |
|:-----|:--------|:---------|
| 层次 | 应用层（消息格式） | 传输层（SSE 协议） |
| 证据 | 逻辑推理 | Python 精确复现 + 原始 SSE 事件流 |
| 可验证性 | 间接（主观推理） | 直接（46 个 SSE 事件，finish_reason 全程为 null） |
| 修复量 | 多个文件，多个函数 | 3 行代码 |

**教训：精度比速度重要。与其猜，不如创造条件做精确验证。**

## 四、实战教训

### 4.1 识别"无前置条件推理"的信号

当你发现自己在以下状态时，说明你在无前置条件推理：

- 反复看同一份代码，希望"再看一遍能找到什么"
- 推理链条很长（5 步以上），但每步都用了"可能""大概""通常"
- 需要不断假设 API/框架/库的行为，但又没有文档或实验确认
- 修改了代码但不确定是否有生效
- 依赖的信号从未验证是否真实存在

### 4.2 应该怎么做

#### 第一步：停下来，写缺失条件

不要继续往下推。把"我现在不知道什么"写下来。

#### 第二步：区分优先级

每个缺失条件问自己：

```
Q1: 这个条件如果拿到，能直接证明/证伪哪个假设？
Q2: 它需要多少成本去获取？（1 分钟？10 分钟？1 小时？）
Q3: 是否有其他条件覆盖了同样的假设？
```

优先级 = 证据力 / 获取成本

#### 第三步：选一个先验证

不要同时追 5 个方向。选优先级最高的，花最少成本去验证。

#### 第四步：验证完如果还不行，换方向

同一个方向试 3 次失败就换（工程铁律第三条）。

### 4.3 本次实战验证的方法

**Python 精确复现法**：把 GDScript 构造的请求体原样复制到 Python 中发送。这种方法的价值在于：
- 确认问题不在 API 侧（API 正常工作）
- 确认问题在客户端侧（我们的代码没正确处理响应）
- 获得原始 SSE 数据（GDScript 的 HTTPClient 没有日志输出）

**附带验证法**：同一请求发 3 次，确认结果可重复。

## 五、AI 协作模式改进建议

### 5.1 给 AI 的规则

1. **当推理链长度 > 5 步且每步都有不确定性时，停下来列前置条件，而不是继续推**
2. **列条件时必须标注优先级**（基于证据强度），不能扁平化列出
3. **优先做低成本精确验证**（Python 复现、抓包、加日志），其次才是逻辑推理
4. **当你觉得"再看一遍代码"时，说明需要新信息，不是需要更多分析**

### 5.2 给人类的协作建议

1. **AI 开始绕圈时，中断它** — 用户做对了这一点（"你不用思考这个问题了"）
2. **告诉 AI 它缺的不是代码分析，是新数据** — 提供原始 SSE 事件流让 AI 做精确判断
3. **用 Python/Rust 等成熟语言做边缘验证** — GDScript 缺少开发工具支持（抓包、日志），用 Python 补位
