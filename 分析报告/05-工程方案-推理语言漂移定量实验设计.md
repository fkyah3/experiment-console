# 工程方案：推理语言漂移定量实验

> 分析层 → 工程层
> 前置条件：实验台已具备真实 tool calling + 多轮能力（V2）

---

## 一、实验目标

对"英文进入对话流"的触发条件做**精确量化**，产出可复用的阈值数据。

### 要回答的问题

1. **第一推需要多少英文量？** — AI 第一次切换 reasoning 语言时，上下文中英文总 token 数/占比
2. **直接相关 vs 累积？** — 是单次注入 5KB 代码就能触发，还是需要多次渐进累积
3. **tool_calls 结构和 tool 返回内容各自贡献多少？** — 函数声明字段（list_dir, filePath）的英文 vs 代码内容的英文
4. **锚定词的效果边界在哪？** — 加上"reasoning 必须使用中文"对触发阈值的影响

---

## 二、实验设计

### 基线流程

所有实验共享同一个基线流程：

```
Round 1: user(zh) → assistant(zh reasoning + tool_calls) → tool(zh 结果)
          ↓
Round 2: assistant(zh reasoning) → user(zh "继续")
          ↓
          确认：纯中文对话环境下 reasoning 保持 zh
```

### 实验组

#### 实验 A：单轮冲击阈值

| 组 | Round 1 tool 返回内容 | 英文量 | 预期 |
|:-|:---------------------|:------|:-----|
| A1 | 中文目录（全中文件路径） | 0% | zh → zh ✅ |
| A2 | 小量英文代码（read 1KB） | ~1KB | zh → zh 或 en？ |
| A3 | 中量英文代码（read 5KB） | ~5KB | zh → zh 或 en？ |
| A4 | 大量英文代码（read 20KB） | ~20KB | zh → zh 或 en？ |
| A5 | 完整 provider.ts（51KB） | ~51KB | zh → en ✅ |

**目的：** 确认单次注入的阈值。如果 A2-A4 全 zh 而 A5 en，说明阈值在 20-51KB 之间；如果 A3 就触发，说明阈值为 ~5KB。

#### 实验 B：累积效应

| 组 | 多轮注入顺序 | 总英文量 | 预期 |
|:-|:------------|:--------|:-----|
| B1 | 第一轮 2KB → 第二轮 3KB → 第三轮 5KB | 10KB | 在哪一轮触发 |
| B2 | 第一轮 5KB → 第二轮 5KB | 10KB | 一次触发 vs 两次触发 |
| B3 | 先 10KB 中文 → 再 10KB 英文 | 10KB | 中文前置能否延迟触发 |

**目的：** 确认 B1 触发总阈值低于 A 组单次阈值的话，说明累积效应存在。

#### 实验 C：锚定词效果量化

对 A1-A5 的每组，分别以：
- **C 系列：** system prompt 不含"reasoning 必须中文"
- **D 系列：** system prompt 含"reasoning 必须中文"

| 组 | 锚定词 | 英文量 | 漂移点 |
|:-|:------|:------|:------|
| C1 | 无 | 5KB | 首轮？ |
| D1 | 有 | 5KB | 首轮？ |
| C2 | 无 | 20KB | 首轮？ |
| D2 | 有 | 20KB | 首轮？ |
| C3 | 无 | 51KB | 首轮？ |
| D3 | 有 | 51KB | 首轮？ |

**目的：** C 系列-D 系列的差值 ≈ 锚定词对 reasoning 层的实际约束力。

## 三、执行方式

### 手动运行（每轮单独发送）

实验台的积木面板已支持手动控制每轮：
1. 编辑积木 → 填入 system + user → 发送
2. 收到 assistant（含 reasoning + tool_calls）→ 手动填入 tool 返回内容
3. 再次发送 → 携带上一轮的 reasoning
4. 重复

### 批量运行（BatchRunner）

每组需要 10-20 轮来消除随机性。BatchRunner 的 `prototype_messages` 对应 Baseline 流程的前几轮，Run 完后看 `_stats.reasoning_language` 分布。

### 观测指标

| 指标 | 来源 |
|:-----|:-----|
| 本轮 reasoning 语言（zh/en） | BatchRunner._stats |
| 本轮 reasoning tokens 数 | usage.completion_tokens_details.reasoning_tokens |
| 本轮输入 tokens | usage.prompt_tokens |
| 上下文中英文 token 占比 | 需手动计算（prompt 中的英文字符/total） |
| 漂移发生时是第几轮 | reasoning_language 从 zh 切到 en 的那轮 |

## 四、预期产出

1. **阈值表格：**

| 注入方式 | 无锚定 | 有锚定 | 差值 |
|:--------|:------|:------|:-----|
| 单次冲击 | X KB | Y KB | Z KB |
| 累积 3 轮 | A KB | B KB | C KB |

2. **累积曲线：** 多轮累积与单次冲击的阈值差异图

3. **锚定词效力：** 有/无锚定词的触发阈值偏移量（Z 值）

4. **工程建议：**
   - 如果锚定词只能把阈值从 5KB 推到 8KB（3KB 偏移），说明需要运行时干预而不仅靠 prompt
   - 如果锚定词能把阈值从 5KB 推到 50KB+（45KB 偏移），说明纯 prompt 方案可行

---

## 五、已知限制

1. **tool_calls 字段的英文是硬编码的** — `filePath`、`dirPath`、`list_dir`、`read` 等函数名本身就是英文，这部分英文无法通过任何方案消除。它是 API 协议的一部分。
2. **实验在 experiment-console 内完成** — 不在 opencode 内部运行。结论在移山架构中实现时需要额外适配。
3. **模型版本可能影响阈值** — 当前使用 deepseek-v4-flash。切到 deepseek-v4-pro 时阈值可能不同。
