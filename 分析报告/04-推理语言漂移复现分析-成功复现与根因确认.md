### 推理语言变化序列

两份实验的 reasoning 过程完全一致：

```
#1  assistant reasoning（zh）
     "用户想分析 opencode 项目的 Provider 系统。我需要先探索项目结构。"
     tool_calls: [list_dir /]

#2  tool result: 目录列表（英文文件名 + 结构信息）
    assistant reasoning（en 🔴！）
     "Let's look at the packages directory since this is a monorepo structure."
     tool_calls: [list_dir packages]

#3  tool result: packages 目录
    assistant reasoning（en）
     "The opencode package seems most relevant. Let's explore it."
     tool_calls: [list_dir packages/opencode]

...后续全 en...
```

**核心观测：** 第一次 tool 返回后，下一次 assistant 的 reasoning 就从中文切到了英文。之后即使读了多轮中文文件名/结构信息，reasoning 不再恢复。

### 关键差异对比

| 维度 | 失败的静态模板 | 成功的真实 tool call |
|:----|:--------------|:-------------------|
| 工具调用的决定者 | **模板预设** | **AI 自己** |
| assistant 消息结构 | `{reasoning, content}` 无 tool_calls | `{reasoning, tool_calls}` 有函数调用 |
| 第一轮 reasoning 来源 | **预设中文文本** | **模型自己产出** |
| 英文代码的角色 | 被塞入的历史消息 | AI 主动请求并收到的"工作成果" |
| 对话流的语言 | 全程中文对话 + 一条工具结果 | **中文 reasoning + 英文工具调用 + 英文目录 → 混合上下文** |

## 三、根因确认：修正后的污染链

### 错误的旧模型

```
英文代码注入 → 上下文英文占比 → 超过阈值 → reasoning 切换
```
（阈值假设：静态模板验证了即使 52KB 也冲不动）

### 正确的模型

```
AI 用中文 reasoning 决定调工具
  → assistant 消息同时有了 {zh reasoning + en tool_calls}
  → tool 返回 en 代码
  → 下一轮请求中，对话流 = zh推理 + en函数声明 + en代码内容
  → reasoning 层感知"当前任务的语言混合了英文"
  → AI 产出英文 reasoning
  → 下一轮请求携带自己的英文 reasoning
  → 自锁循环：en reasoning → 更多 en reasoning（99.4% 不可逆）
```

### 一句话根因

**漂移需要英文进入对话流（assistant 消息的 tool_calls/function 层），而不是工作材料（tool 返回的内容层）。** 静态模板把英文全部放在了"工作材料"位置，对话流全程中文——即使 52KB 也冲不动。真实 tool calling 让英文代码的函数声明、参数名进入了对话流的 assistant 消息，触发了语言切换。

### 锚定词的效果

"reasoning 和回复都必须使用中文"——**仍然触发了漂移**。这说明 system prompt 对 reasoning 层的控制力确实弱于对 content 层的控制力。

## 四、两种 reasoning 漂移的最终定论

| | Session 1（工程层背景阅读） | 本次实验 |
|:--|:--------------------------|:---------|
| 触发源 | read 返回 51KB 英文 TypeScript | list_dir 返回英文目录 + 后续 read 英文文件 |
| 触发机制 | tool_calls + tool 返回混合 | tool_calls + tool 返回混合（相同） |
| 自锁 | 99.4% 不可逆（3085/3103 条 en） | 持续 en（与 Session 1 一致） |
| content 层 | 保持中文 ✅ | 保持中文 ✅ |

**确认：漂移机制不变。实验台成功复现了真实场景的完整链路。**

## 五、工程意义

1. **实验台可以用于精确定位触发条件**——每次 tool call 作为一轮，我们可以精确控制每一轮注入的英文量，找到"第一推"的阈值
2. **锚定词的效果已量化**——加上"reasoning 必须中文"仍然触发，说明需要更强的方案
3. **content 层始终不受影响**——确认了 system prompt 牢靠地锁住了输出层
4. **累积效应已验证**——实验台实现的多轮机制和真实 opencode 的逐轮累积一致
