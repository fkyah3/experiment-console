# 语言层实验总结：用精准的 token 描述需求，而非为省而省

> 2026-05-15
> 编写：语言层
> 核心观点：我们的目的是利用 AI 产出与需求实际相符的产物。不是为了省 token 而省 token。如果为了省 token 而省，还不如什么都不做，或者不用 AI。

---

## 一、我们做了什么

从 "过度精确" 理论的推导开始，到 ExperimentConsole 上线后，累计完成了 **数十轮实验，总计 ~1200 次 API 调用**。覆盖以下维度：

| 维度 | 实验内容 | 调用量 |
|:----|:--------|:------|
| Prompt 措辞迭代 | 4 代 × 多轮 batch，找稳定性最高的语言锚定 | ~500 |
| 推理强度对比 | Flash/Pro, effort high/max, 中英文问题 | ~320 |
| 多轮工具调用漂移 | 3 组 prompt × 10 轮多轮任务对比 | ~300 |
| 多语言推理 | 中文/日语/波斯语/俄语/西语/韩语/文言文/昆雅语 | ~200 |
| 全量数据验证 | 无 prompt 对照组、分离批量实验 | ~80 |

## 二、核心结论

### 结论 1：省 token 不是目标，精准才是

实验中观察到的最核心矛盾：

| 做法 | 现象 | 结果 |
|:----|:----|:-----|
| 盲目压 token | prompt 从三行压到一行，丢掉"如果不翻译"约束 | 英文代码被翻译，输出质量下降 |
| 为了省 rt 加"缜密" | 目的是少想，结果触发了过度规划 | 一个 outlier 花了 789 rt |
| 为了省 duration 关 thinking | 速度快了，但复杂推理能力丧失 | 不同场景需不同方案 |
| **精准描述需求** | "你的 reasoning 和 content 都必须使用中文 | 代码、路径等不翻译" | **avg rt 从 236 降到 50，且稳定性从 40% 拉到 100%** |

最简单的版本（"你是一个 AI 工程助手"，不加任何语言锚定）看起来 token 最少、prompt 最省，但实际结果是什么？中文 reasoning 只有 30%。省了 30 个 prompt token，丢掉了 70% 的中文稳定性。

**prompt 的长度不重要，每个 token 是否在精准传达需求才重要。**

### 结论 2：中文是目前最稳定的推理语言

| 语言 | 稳定性 | avg rt | 说明 |
|:----|:------:|:------:|:-----|
| **中文（口语）** | **100%** | **304** | 母语级推理，无漂移 |
| 日语 | ~90% | 1054 | 本地化可用，但 token 成本是中文的 3.5 倍 |
| 波斯语 | 92% | 471 | 本地化可用，有随机英文漂移 |
| 文言文 | ≈0%（退回到现代中文做 reasoning） | 229 | 有完整词汇体系，但缺乏"用文言文推理"的训练模式 |
| 昆雅语 | ≈0%（退回到英文） | — | 词汇和体系都不完整 |

中文在稳定性上持续最优，在效率上也最优（avg rt 304 vs 日语的 1054）。**如果你不要求输出本地化到其他语言，中文就是最优选择。**

日语虽然 rt 接近中文的 3.5 倍（1054 vs 304），但那 1054 不是浪费——是模型在用日语做完整的深度推理，产出高质量的日语技术摘要。如果用户需要日语输出，这 1054 rt 是必要的成本。如果想要省 token 把日语 prompt 去掉，用户得到的就不是想要的日语答案。

**这不是"日语效率低"，是"日语的推理路径在训练数据中的覆盖度不如中文完整"，导致表达同样内容需要更多推理步骤。这是一个客观的模型限制，不是可以优化的。** 接受它，或者选择中文作为默认工作语言。

### 结论 3：在多轮工具调用场景下，prompt 的天花板已到

三组对比（基准 prompt、"理科高中生"身份锚定、B站式机械标记）的结果一致：

| 方案 | 中文 rt | duration | 说明 |
|:----|:-------:|:--------:|:-----|
| 基准 prompt | 30-50% | 62.7s | 无额外成本，但稳定性低 |
| 身份锚定 | ~50% | 66.2s | 几乎不影响 |
| B站机械标记 | 90% | 69.1s | 用 3x rt 换 20% 稳定性提升，但速度和基线无本质差异 |

分析层的一个关键发现证实了根因：**V4 论文提到 Interleaved Thinking 将 reasoning 历史改为永久保留。一旦漂移就形成正反馈——越漂越多，越难回来。** 这不是 prompt 能压住的，是架构级决定。

**真正提升稳定性的方案不在 prompt 层，在工程层。**

### 结论 4：多语言实验的底层规律

语言锚定的稳定性与训练数据占比强相关，与 prompt 的措辞好坏弱相关：

| 训练数据覆盖 | 可达到的稳定性 | 代表语言 |
|:------------|:--------------|:--------|
| 极高（公司母语） | 100% | 中文 |
| 高（成熟市场） | 90-95% | 日语、西班牙语、俄语 |
| 中低（稀疏覆盖） | 85-92% | 波斯语 |
| 接近于零 | ≈0% | 昆雅语、文言文（推理层） |

这不是 prompt 工程能突破的规律。这是模型在训练阶段建立的推理路径决定的。

### 结论 5：不锚定一个版本可能比锚定错误更好

"你是一个 AI 工程助手"（无语言锚定）vs 锚定版 prompt 的对比：

| 条件 | 中文 rt | 输出语言 |
|:----|:------:|:--------|
| 不锚定 | ~30-40% | 100% 中文 content |
| 锚定版 | 100%（单轮） | 100% 中文 content |

两者的 output 都能保证中文。差异只在 reasoning 语言。**如果用户不关心 reasoning 内容，不锚定可能比锚定了一个不稳定的版本更好——因为不锚定至少不会触发"为了遵守承诺而多花 token"的元思考。**

## 三、对移山的实际建议

### 1. Prompt 用最终锁定版

```
你的 reasoning 和 content 都必须使用中文
代码、路径、工具名、文件名、错误信息不翻译
```

不需要再加额外的"决策果断""缜密"等修饰词。实验数据证明它们不产生可衡量的收益。

### 2. 明确区分"省 token"和"精准描述"的场景

| 场景 | 建议 | 理由 |
|:----|:-----|:-----|
| 简单概念解释 | 上述 prompt + thinking disabled | 不需要深度推理，关 thinking 更省 |
| 复杂技术分析 | 上述 prompt + thinking enabled | 需要深度推理，多花 rt 获得质量 |
| 多轮工具调用 | 工程层做上下文管理 | prompt 已到天花板，瓶颈在 tool wait |
| 需要本地化输出 | 目标语言 prompt | 接受 1.5-3.5x rt 的成本 |
| 内部使用，不看 reasoning | 不锚定也可以 | reasoning 语言不影响 content 质量 |

### 3. 工程层优先级高于 prompt 层

数据表明，对用户体验影响最大的不是 "用哪条 prompt"，而是：

| 工程问题 | 对体验的影响 |
|:---------|:------------|
| tool call 等待时间 | 单轮 62s → 分离后 37s |
| 代码预先加载（分离批量） | 漂移从 40% 降到 0% |
| 上下文管理（修剪 reasoning 历史） | 防止 V4 正反馈漂移恶化 |

**prompt 能让单轮场景达到最优。多轮场景需要工程层的介入。**

## 四、一些假设（实验数据无法直接验证的）

### 假设 1：temperature=1 是所有漂移的根因

thinking 模式下 temperature 固定为 1，意味着每条 reasoning token 的生成都是概率采样，而非确定性选最优。从数据上看，同一条件的批量实验总会有 5-10% 的漂移率，与训练数据覆盖度反相关。**这合乎"高温度 → 采样噪声"的解释，但无法通过实验台数据直接证明。** 需要 DeepSeek 官方确认 thinking 模式下 temperature 是否确实被固定为 1，或者用户设置的温度在特定条件下是否生效。

### 假设 2：机房不同节点放大噪声

日语 50 次中出现一次 4096 max out 的极端情况，而同语言其他请求正常。这种"偶发极端值"在不同语言、不同实验批次中都被观测到。**猜这是不同 inference node 的硬件差异（KV Cache 状态、浮点精度）导致的，但没有跨节点实验数据支撑。**

### 假设 3：文言文在 content 层的表现也可能没那么好

从实验的 output 看，文言文 content 在语法和用词上做到了表面的"文言化"。但对于更专业的任务（如代码注释、API 文档），缺乏"技术文言文"的训练数据可能导致 output 出现语义偏差。**本次实验只做了摘要任务，没有测试文言文在代码场景下的实际表现。**

## 五、一路以来的发现轨迹

```
过度精确理论（亲属词的多余维度触发不必要推理）
  ↓
Prompt 措辞直接影响 reasoning token（13 倍差异）
  ↓
identity 锚定（"你是中国人"）比 instruction 锚定（"请用中文思维"）稳定
  ↓
"都"字 + "不翻译" 锁定最终版 prompt
  ↓
多轮场景下 prompt 天花板已到 → 任务框架效应
  ↓
V4 论文确认 Interleaved Thinking 使漂移恶化
  ↓
多语言实验：训练数据占比决定稳定性下限
  ↓
文言文与昆雅语同等价值（推理层无效）
  ↓
省 token 不是目标。用精准的 token 描述需求才是。
```
